================================================================================
1. Назначение проекта
================================================================================

Dispatcher — универсальная диспетчерская / SCADA-платформа для опроса,
мониторинга, управления, визуализации и анализа состояния оборудования,
объектов, систем и технологических процессов.

Платформа не должна быть привязана к одному конкретному объекту, протоколу,
типу оборудования или отрасли. Она должна работать как универсальный
конструктор диспетчеризации.

Основа системы:

    объект -> устройство -> тег -> значение -> качество -> история -> событие
    -> авария -> виджет / мнемосхема / дашборд

Ключевая идея: теговая модель является центральной моделью данных.

================================================================================
2. Основные цели Dispatcher
================================================================================

Dispatcher должен обеспечивать:

1. Опрос оборудования по промышленным и сетевым протоколам.
2. Хранение текущих значений тегов в памяти.
3. Архивирование истории значений по настраиваемым политикам.
4. Контроль качества данных.
5. Обнаружение, регистрацию и обработку аварий.
6. Единый диспетчер событий.
7. Безопасное выполнение команд управления.
8. Визуализацию через дашборды, виджеты и мнемосхемы.
9. Панельные дашборды для локальных экранов и промышленных панелей.
10. Рабочие места операторов и учет активной смены.
11. Конфигуратор оборудования для добавления, импорта, проверки и интеграции
    устройств.
12. Системную диагностику платформы.
13. Пользователей, роли, права, сервисные аккаунты и аудит.
14. Уведомления, планировщик, скрипты, ППР и интеграции как расширяемые
    подсистемы.
15. Эксплуатацию на Linux.
16. Разработку на Windows в Visual Studio с CMake.
17. Возможность будущего перехода части модулей в микросервисы.

3. Технологический стек
================================================================================

Backend:

    Язык: C++20 минимум, C++23 по мере необходимости.
    Система сборки: CMake.
    Среда разработки: Visual Studio 2026.
    ОС разработки: Windows.
    ОС эксплуатации: Linux.
    Управление зависимостями: vcpkg.
    HTTP / WebSocket backend: фреймворк с поддержкой REST и WebSocket,
    кандидат — Drogon или аналог после проверки на этапе каркаса.
    Сеть / async I/O: Boost.Asio.
    Логирование: spdlog.
    Форматирование: fmt.
    JSON: nlohmann/json или simdjson по необходимости.
    База данных: PostgreSQL.
    Тесты C++: GoogleTest.
    Будущий внутренний RPC: gRPC / Protobuf, если потребуется.
    Будущая событийная шина: NATS / NATS JetStream, если потребуется.

Frontend:

    Blazor WebAssembly.
    UI-библиотека для MVP: MudBlazor.
    Сложные SCADA-компоненты: собственные компоненты.
    Мнемосхемы: SVG + собственная модель элементов.
    Графики: отдельная библиотека через JS interop или собственные компоненты.

Протоколы MVP:

    MockDriver.
    SimulatorDriver.
    Modbus TCP.
    SNMP basic.

Сеть:

    IPv4 поддерживается обязательно.
    IPv6 поддерживается обязательно.
    DNS-имена поддерживаются обязательно.

Docker:

    Docker не используется на первом этапе.
    Архитектура не должна мешать добавить Docker позже.

4. Архитектурный стиль
================================================================================

Выбранная архитектура: модульная сервисная архитектура.

На первом этапе все backend-модули запускаются внутри одного приложения:

    backend/apps/dispatcher_server

Каждый крупный функциональный блок оформляется как отдельная библиотека /
модуль со своими:

    интерфейсами;
    моделями;
    тестами;
    документацией SERVICE.md;
    границами ответственности.

В будущем отдельные модули могут быть вынесены в самостоятельные сервисы без
полной переработки бизнес-логики.

Основные режимы модулей в будущем:

    Embedded mode — модуль работает внутри dispatcher_server.
    External service mode — модуль работает как отдельный сервис.

Кандидаты на будущий вынос в микросервисы:

    Polling Service;
    Historian Service;
    Notification Service;
    Integration Service;
    Script Service;
    Scheduler Service;
    Report Service;
    Diagnostics Service.

Скорее остаются в ядре:

    Core;
    Settings;
    Configuration;
    Object Model;
    Devices;
    Tags;
    Security;
    Access Control;
    Dashboards;
    Mimics;
    Audit.

7. Объектная модель
================================================================================

Объекты хранятся в универсальном дереве:

    ObjectNode

Типы объектов:

    Site;
    Building;
    Floor;
    Room;
    Zone;
    Cabinet;
    Rack;
    Line;
    EquipmentGroup;
    Custom.

Устройства привязываются к ObjectNode через location_object_id.

В событиях и авариях сохраняется snapshot названия и местоположения, чтобы
история оставалась понятной даже после изменения объектной модели.

Права на объекты наследуются вниз по дереву.

================================================================================
8. Оборудование и Конфигуратор оборудования
================================================================================

Добавление, импорт, проверка и интеграция устройств выполняются через
Конфигуратор оборудования.

Отдельный сервис DeviceOnboarding не создается.

Импорт устройств, подключений, тегов и адресов не пишет напрямую в рабочую
конфигурацию.

Импортированные данные сначала появляются как черновики в Конфигураторе
оборудования:

    device drafts;
    draft connections;
    draft tags;
    draft commands;
    draft alarm rules.

Пользователь должен иметь возможность:

    отредактировать черновики;
    проверить настройки;
    выполнить проверку связи;
    выполнить тестовый опрос;
    удалить лишнее;
    интегрировать выбранные устройства в рабочую систему.

Конфигуратор не позволяет интегрировать устройство с критическими ошибками.

До интеграции устройство не участвует в:

    опросе;
    авариях;
    истории;
    командах;
    уведомлениях;
    дашбордах.

================================================================================
9. Протоколы и драйверы
================================================================================

Все протоколы реализуются через единый интерфейс:

    IProtocolDriver

Драйвер должен уметь:

    читать значения;
    выполнять batch-read;
    писать значения, если протокол поддерживает запись;
    проверять связь;
    выполнять тестовый опрос;
    отдавать capabilities;
    отдавать schema настроек для UI;
    возвращать подробный ProtocolReadResult / ProtocolWriteResult.

Драйвер не пишет напрямую в БД.

Стартовые драйверы:

    MockDriver;
    SimulatorDriver;
    ModbusTcpDriver;
    SnmpDriver.

Порядок разработки:

    IProtocolDriver;
    MockDriver;
    SimulatorDriver;
    Polling Service;
    Modbus TCP;
    SNMP.

Проверка связи комбинированная:

    ping, если доступен;
    проверка TCP/UDP порта;
    протокольная проверка;
    тестовое чтение.

Тестовый опрос не пишет значения в TagValueStore и не пишет историю.
Он является диагностической операцией и возвращает TestPollResult в UI.

10. Теги и текущие значения
================================================================================

Тег — центральная сущность Dispatcher.

Типы тегов:

    DeviceTag;
    ManualTag;
    CalculatedTag;
    SystemTag;
    VirtualTag;
    CommandTag;
    SimulationTag;
    ExternalTag.

Текущее значение хранится в памяти:

    TagValueStore

БД не является основным источником текущих live-значений.

Текущие значения могут сохраняться в БД как snapshot для восстановления после
рестарта, но live-логика должна работать через память.

TagCurrentValue включает:

    tag_id;
    value;
    raw_value;
    engineering_value;
    quality;
    timestamp;
    source_timestamp;
    server_timestamp;
    last_good_value;
    last_good_timestamp;
    change_counter;
    source.

Качество данных участвует во всей логике:

    отображение;
    история;
    аварии;
    команды;
    скрипты;
    уведомления.

Поддерживаемые качества:

    Good;
    Uncertain;
    Bad;
    CommunicationError;
    Timeout;
    DeviceError;
    ProtocolError;
    OutOfRange;
    Manual;
    Simulation;
    Maintenance;
    Stale;
    Disabled;
    NotInitialized.

Поддержка value extraction обязательна:

    bit extraction;
    byte extraction;
    mask;
    shift;
    bit length;
    enum mapping;
    boolean mapping.

================================================================================
11. Опрос оборудования
================================================================================

Polling Service должен быть многопоточным / асинхронным, но не должен создавать
постоянный отдельный поток на каждое устройство.

Модель:

    PollingScheduler -> PollingQueue -> WorkerPool / async I/O
    -> ProtocolDriver -> TagValueStore -> EventBus

Ограничения:

    limit по устройству;
    limit по подключению;
    limit по протоколу;
    limit по PollingNode;
    rate limit;
    backoff policy;
    retries;
    timeout.

Частота опроса настраивается:

    на уровне устройства;
    на уровне группы тегов;
    на уровне тега.

Нужно поддержать частоту от 0,5 секунды и выше, с диагностикой нагрузки.

================================================================================
12. История
================================================================================

История значений пишется только по TagArchivePolicy.

Поток:

    TagValueStore
    -> ArchivePolicyEvaluator
    -> HistorianBuffer
    -> BatchWriter
    -> PostgreSQL

TagArchivePolicy:

    Disabled;
    OnChange;
    Periodic;
    OnChangeWithDeadband;
    OnQualityChange;
    OnAlarm;
    AlwaysButThrottled.

Для каждого тега:

    archive_enabled;
    archive_mode;
    archive_period_ms;
    archive_deadband;
    archive_on_quality_change;
    archive_on_alarm;
    archive_priority;
    min_archive_interval_ms.

При перегрузке Historian:

    Critical сохранять первыми;
    Important сохранять по возможности;
    Normal прореживать;
    low/service можно сбрасывать;
    обязательно создавать событие HistorianOverloaded.

База: PostgreSQL.
TimescaleDB — возможно позже, но не обязательная зависимость MVP.

13. Аварии и события
================================================================================

Авария — это особый тип события с жизненным циклом.

Источники аварий:

    теги;
    связь;
    узлы опроса;
    скрипты;
    система;
    интеграции;
    ручные события;
    ППР.

Все переходы аварии попадают в единый Диспетчер событий.

Единый Event Service должен хранить:

    аварии;
    команды;
    действия пользователей;
    действия панелей;
    действия рабочих мест;
    события связи;
    изменения конфигурации;
    скрипты;
    планировщик;
    ППР;
    интеграции;
    системные события;
    события безопасности.

Команды и аварии связываются через correlation_id / causation_id.

================================================================================
14. Команды и управление
================================================================================

Прямой записи в тег через UI API быть не должно.

Любое управление оборудованием выполняется только через:

    Command Service

Поток:

    UI / Script / Scheduler / API
    -> Command Service
    -> AccessControl / CommandPolicy
    -> CommandQueue
    -> ProtocolDriver / PollingNode
    -> Result Verification
    -> Audit
    -> EventBus

Команда имеет:

    код;
    имя;
    описание;
    target_device_id;
    target_tag_id;
    command_type;
    parameters;
    risk_level;
    confirmation_policy;
    permission_policy;
    execution_policy.

Risk levels:

    Low;
    Medium;
    High;
    Critical.

Нужно поддержать:

    confirmation;
    reason/comment по настройке;
    повторную авторизацию по настройке;
    CommandLock;
    очереди по устройству/подключению/узлу;
    DryRun;
    Simulation;
    RealExecution;
    аудит всех попыток.

Dual approval заложить, но реализовать позже.

================================================================================
17. Пользователи, роли и безопасность
================================================================================

Типы аккаунтов:

    User;
    ServiceAccount;
    WorkstationAccount;
    PanelServiceAccount;
    ExternalUser;
    SystemAccount.

Права:

    Role + Permission + Scope.

Права наследуются по объектному дереву.

Отрицательные права / explicit deny не используются.

Отдельного права read_tag не делаем.
Если пользователь имеет доступ к объекту, дашборду, рабочему месту или панели,
он видит теги, которые там отображаются.

Управление проверяется через:

    CommandPolicy;
    право на команду;
    объект;
    источник команды;
    режим клиента;
    активную смену, если требуется;
    качество данных;
    состояние устройства;
    ControlBlock;
    risk level;
    confirmation policy.

Пароли и токены не хранятся в открытом виде.
Токены хранятся только как hash.

MFA, AD, LDAP, OIDC, SAML — заложить, реализовать позже.

18. API и WebSocket
================================================================================

REST API используется для:

    начальной загрузки;
    конфигурации;
    команд;
    истории;
    администрирования;
    интеграций.

WebSocket используется для live-обновлений:

    текущие значения;
    качество;
    аварии;
    события;
    команды;
    состояние устройств;
    состояние системы;
    уведомления;
    панели;
    рабочие места.

Основные группы API:

    /api/v1/ui
    /api/v1/admin
    /api/v1/integration
    /api/v1/panel
    /api/v1/workstation
    /api/v1/system
    /api/health

WebSocket работает через управляемые подписки активного экрана.
Клиент подписывается только на то, что реально отображает.

REST initial load + WebSocket updates — основной подход для frontend.

Во всех важных операциях используется correlation_id.

================================================================================
19. Системные настройки
================================================================================

Все системные настройки, которые можно безопасно менять во время работы,
должны быть доступны через веб-интерфейс администратора.

Файлы конфигурации используются только для bootstrap-настроек:

    строка подключения к PostgreSQL;
    порт запуска backend;
    первичный master key / путь к секретам;
    режим первого запуска;
    базовый путь к файлам;
    минимальные параметры логирования до старта БД.

Рабочие настройки хранятся в БД.

Настройки должны иметь:

    код;
    название;
    описание;
    тип;
    значение по умолчанию;
    допустимый диапазон;
    область применения;
    уровень доступа;
    необходимость перезапуска;
    возможность hot reload;
    признак секретности;
    аудит изменений.

22. MVP
================================================================================

MVP — это первый полный рабочий контур диспетчерской системы, а не набор
отдельных демонстрационных функций.

MVP должен включать:

    объектную модель;
    устройства;
    теги;
    Modbus TCP;
    SNMP basic;
    IPv4 / IPv6;
    Конфигуратор оборудования;
    проверку связи;
    тестовый опрос;
    Polling Service;
    TagValueStore;
    WebSocket updates;
    историю по политике;
    простые аварии;
    Диспетчер событий;
    базовые команды;
    аудит;
    базовые роли;
    простые дашборды;
    PanelDashboard basic;
    Workstation Mode basic;
    мнемосхемы basic;
    системную диагностику;
    системные настройки через UI;
    backup basic;
    Linux deploy через systemd.

Не входит в MVP:

    микросервисы;
    удаленные PollingNode;
    TimescaleDB как обязательная зависимость;
    полный редактор мнемосхем;
    полный drag-and-drop редактор дашбордов;
    полный Script Editor;
    Script Debugger;
    Rule Engine без кода;
    MFA;
    dual approval;
    AD / LDAP / OIDC / SAML;
    сложная эскалация уведомлений;
    SMS / Telegram / Teams;
    OPC UA / MQTT / BACnet / S7 / IEC 104 / DNP3;
    ServiceDesk / ERP / 1C / CMMS;
    полный report designer;
    hot standby;
    кластеризация;
    мобильное приложение;
    Docker deployment.

24. Правила взаимодействия с ИИ-помощником
================================================================================

Работа выполняется пошагово.

Каждый шаг:

    1. Краткое объяснение, что делаем.
    2. Команды терминала для создания папок/файлов.
    3. Код или содержимое файлов.
    4. Команды запуска проекта и тестов.
    5. Что проверить вручную.
    6. Команды git commit и git push.
    7. Ключевая фраза для перехода к следующему шагу.

Команды терминала группируются по смыслу в один блок.
Каждая команда должна быть одной строкой.

Если файл нужно сильно изменить, файл переписывается полностью.
Если нужно добавить несколько строк, указывается точное место вставки.

Каждый шаг завершается commit + push в ветку master.

Если возникла ошибка, пользователь отправляет ошибку в чат.
Если все прошло успешно, пользователь отправляет ключевую фразу шага.

25. Важные текущие договоренности
================================================================================

Репозиторий называется dispatcherV1, потому что dispatcher уже занят.

Внутри проекта используем имя продукта:

    Dispatcher

Внутренние имена не должны использовать V1.

Используем только x64-debug и x64-release.

x86 не используем.

Ninja не используем, так как он вызывает проблемы в Visual Studio.

Работаем через терминал Visual Studio.

Папка out не хранится в репозитории.
