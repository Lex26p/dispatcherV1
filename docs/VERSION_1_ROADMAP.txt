================================================================================
1. Назначение документа
================================================================================

Этот документ фиксирует дорожную карту первой версии Dispatcher.

В отличие от DEFERRED_FEATURES.txt, этот документ описывает не отложенные идеи,
а практическую очередность разработки первой версии приложения.

Главный принцип:

    не разрабатывать все сервисы одновременно.

После подготовки фундамента сервисы и модули разрабатываются последовательно,
через понятные вертикальные срезы.

Каждый этап должен давать проверяемый результат:

    проект собирается;
    приложение запускается;
    новая часть изолирована и документирована;
    критическая логика покрыта тестами, если применимо;
    SERVICE.md / MODULE.md обновлены;
    ограничения зафиксированы.

================================================================================
2. Цель первой версии
================================================================================

Первая версия Dispatcher должна стать первым рабочим контуром универсальной
диспетчерской системы.

Первая версия должна уметь:

    создавать объектную модель;
    добавлять устройства через Конфигуратор оборудования;
    создавать теги;
    опрашивать симулятор;
    опрашивать Modbus TCP;
    опрашивать SNMP basic;
    хранить текущие значения в памяти;
    показывать текущие значения во frontend;
    отправлять live-обновления через WebSocket;
    архивировать историю по политикам;
    создавать простые аварии;
    показывать события и аварии;
    выполнять базовые команды через Command Service;
    показывать простые дашборды;
    показывать базовые мнемосхемы с zoom колесом мыши;
    поддерживать панельный дашборд basic;
    поддерживать рабочее место basic;
    вести аудит важных действий;
    иметь базовые роли;
    иметь системную диагностику;
    иметь системные настройки через UI;
    иметь basic backup конфигурации;
    запускаться на Linux через systemd.

================================================================================
3. Что не является целью первой версии
================================================================================

Первая версия не должна включать:

    микросервисы;
    удаленные PollingNode;
    обязательный TimescaleDB;
    полный drag-and-drop редактор дашбордов;
    полный редактор мнемосхем;
    Script Debugger;
    Rule Engine без кода;
    MFA;
    dual approval;
    AD / LDAP / OIDC / SAML;
    сложную эскалацию уведомлений;
    SMS / Telegram / Teams;
    OPC UA / MQTT / BACnet / S7 / IEC 104 / DNP3;
    ServiceDesk / ERP / 1C / CMMS;
    полный report designer;
    hot standby;
    кластеризацию;
    мобильное приложение;
    Docker deployment.

Эти функции перечислены в docs/DEFERRED_FEATURES.txt.

================================================================================
4. Общий подход к разработке первой версии
================================================================================

Разработка первой версии идет не по принципу "сначала все модели, потом весь UI",
а через последовательность фундаментальных этапов и вертикальных срезов.

Правильный порядок:

    1. документация и правила проекта;
    2. CMake и структура репозитория;
    3. базовые модули;
    4. common/core/contracts;
    5. базовые доменные модели;
    6. интерфейс протокольного драйвера;
    7. симулятор;
    8. текущие значения;
    9. polling;
    10. REST API;
    11. frontend current values;
    12. WebSocket;
    13. устройства и Конфигуратор оборудования;
    14. реальные протоколы;
    15. история;
    16. события и аварии;
    17. команды;
    18. дашборды;
    19. мнемосхемы;
    20. панели и рабочие места;
    21. безопасность и аудит;
    22. диагностика и настройки;
    23. backup;
    24. Linux deploy;
    25. стабилизация первой версии.

    PollingScheduler
    -> PollingQueue
