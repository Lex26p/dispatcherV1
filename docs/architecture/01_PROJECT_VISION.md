# Dispatcher Project Vision

## 1. Общая информация

## Название проекта

Dispatcher

---

## Назначение

Dispatcher — это промышленная программная платформа класса SCADA/IIoT для мониторинга, управления и анализа инженерных систем.

Основная задача системы:

Создать универсальную платформу, которая позволяет подключать различное оборудование, создавать цифровую модель объекта, получать данные, отображать их оператору, хранить историю и выполнять управление оборудованием.

---

# 2. Основная идея проекта

Dispatcher строится вокруг цифровой модели реального объекта.

Главная концепция:

```
Физический объект

        ↓

Цифровая модель

        ↓

Данные

        ↓

События

        ↓

Анализ

        ↓

Управление
```

Система должна понимать не только отдельные значения датчиков, но и структуру объекта:

```
Предприятие

 └── Площадка

      └── Зона

           └── Оборудование

                └── Устройство

                     └── Параметр
```

---

# 3. Назначение системы

Dispatcher предназначен для:

- автоматизации диспетчерского контроля;
- мониторинга инженерных систем;
- управления оборудованием;
- анализа состояния объектов;
- хранения технологической истории;
- обработки аварийных ситуаций;
- интеграции с внешними системами.

---

---

# 5. Целевая архитектурная модель

Dispatcher проектируется как:

## Модульный промышленный сервер

Основной принцип:

```
Один сервер

+

четко разделенные модули

+

контракты взаимодействия
```

Не используется:

- хаотичная монолитная архитектура;
- сильная связанность модулей;
- прямой доступ модулей к внутренним данным друг друга.

---

Основные задачи:

- управление пользователями;
- права доступа;
- системные настройки;
- обслуживание системы.

---

## Руководитель

Основные задачи:

- отчеты;
- анализ;
- статистика.

# 8. Первый релиз Dispatcher V1

Цель первой версии:

Создать рабочую промышленную платформу с базовыми возможностями.

Основной сценарий:

```
Инженер

↓

Создает объект

↓

Добавляет устройство

↓

Проверяет связь

↓

Создает параметры

↓

Dispatcher получает данные

↓

Оператор видит состояние

↓

Система сохраняет историю

↓

Создаются события и аварии
```

---

# 9. Ограничения первой версии

В первую версию не входят:

- сложные распределенные кластеры;
- облачная платформа;
- машинное обучение;
- сложная аналитика;
- полноценная MES;
- ERP функции.

Архитектура должна позволять добавить их позже.

---

# 10. Главные архитектурные принципы

## Разделение ответственности

Каждый модуль отвечает только за свою область.

---

## Конфигурация через систему

Изменения выполняются через конфигуратор.

Не допускается:

- ручное изменение базы;
- скрытые настройки;
- изменение файлов конфигурации оператором.

---

## Версионность

Все важные изменения должны иметь историю.

Например:

```
Configuration v1

Configuration v2

Configuration v3
```

---

## Аудит

Система должна знать:

- кто изменил;
- когда изменил;
- что изменил;
- результат операции.

---

## Расширяемость

Новые возможности должны добавляться через модули.

---
