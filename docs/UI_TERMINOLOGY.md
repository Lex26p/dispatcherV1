# Dispatcher UI Terminology

## Назначение документа

Этот документ фиксирует терминологию пользовательского интерфейса Dispatcher.

Цель — сделать frontend понятным, единообразным и русскоязычным по умолчанию.

---

# Основное правило

Пользовательский интерфейс Dispatcher пишется на русском языке.

Английский язык рассматривается как будущая локализация.

Правило для дальнейшей разработки:

    новый пользовательский текст во frontend пишем на русском языке

---

# Что считается пользовательским текстом

Пользовательский текст — это все, что видит оператор или администратор в интерфейсе.

К пользовательскому тексту относятся:

- заголовки страниц;
- пункты меню;
- подписи карточек;
- подписи таблиц;
- названия колонок;
- кнопки;
- подсказки;
- предупреждения;
- сообщения об ошибках;
- сообщения загрузки;
- пустые состояния;
- поясняющий текст;
- статусы, отображаемые в UI.

---

# Что не переводим

Не переводим технические элементы:

- имена файлов;
- имена классов;
- имена методов;
- namespace;
- route paths;
- backend module codes;
- API paths;
- internal IDs;
- enum names в коде;
- commit messages;
- protocol names;
- product name Dispatcher.

---

# Название продукта

Product name:

    Dispatcher

На русском в UI можно писать:

    Dispatcher

Не использовать:

- Диспетчер;
- Диспетчерская;
- SCADA Dispatcher RU;
- DispatcherV1.

Причина:

- Dispatcher — название продукта;
- слово не должно меняться по языкам;
- репозиторий может называться `dispatcherV1`, но продукт внутри называется `Dispatcher`.

# Основные доменные термины

## Object

Использовать:

    Объект

Примеры:

- Объекты;
- Дерево объектов;
- Карточка объекта;
- Путь объекта;
- Тип объекта.

Не использовать:

- Нода;
- Узел;
- Object;
- ОбъектNode.

Исключение:

- в технической документации backend можно использовать `ObjectNode`.

## Device

Использовать:

    Устройство

Примеры:

- Устройства;
- Список устройств;
- Состояние устройства;
- Диагностика устройства;
- Подключение устройства.

Не использовать:

- Девайс;
- Device;
- Прибор, если речь не о конкретном физическом приборе.

## Tag

Использовать:

    Тег

Примеры:

- Теги;
- Список тегов;
- Значение тега;
- Адрес тега;
- Тип тега;
- Качество тега.

Не использовать:

- Метка;
- Сигнал, если речь именно о tag model;
- Tag.

Пояснение:

- в SCADA-контексте термин «тег» допустим и понятен;
- «сигнал» можно использовать как бизнес-синоним позже, но базовая модель называется «тег».

## Value

Использовать:

    Значение

Примеры:

- Текущее значение;
- Последнее значение;
- Инженерное значение;
- Исходное значение;
- Последнее достоверное значение.

## Quality

Использовать:

    Качество

Примеры:

- Качество;
- Качество значения;
- Хорошее качество;
- Ошибка связи;
- Недостоверное значение.

Для отображения quality statuses:

- Good — Хорошее;
- Uncertain — Сомнительное;
- Bad — Плохое;
- CommunicationError — Ошибка связи;
- Timeout — Тайм-аут;
- DeviceError — Ошибка устройства;
- ProtocolError — Ошибка протокола;
- OutOfRange — Вне диапазона;
- Manual — Ручное;
- Simulation — Симуляция;
- Maintenance — Обслуживание;
- Stale — Устарело;
- Disabled — Отключено;
- NotInitialized — Не инициализировано.

# Alarms terminology

## Alarm

Использовать:

    Авария

Примеры:

- Аварии;
- Активные аварии;
- Журнал аварий;
- Правила аварий;
- Состояние аварии.

Не использовать:

- Alarm;
- Алерт;
- Тревога.

Пояснение:

- для SCADA/диспетчеризации термин «авария» ближе к промышленному контексту;
- «тревога» может быть использована позже для security-сценариев, но базовый alarm model отображаем как «авария».

## Active alarms

Использовать:

    Активные аварии

## Alarm acknowledgement

Использовать:

    Квитирование

Примеры:

- Требуется квитирование;
- Квитировано;
- Квитировать аварию.

Не использовать:

- Ack;
- Acknowledgement;
- Подтверждение, если речь именно об alarm lifecycle.

## Severity

Использовать:

    Важность

Примеры:

- Важность аварии;
- Важность события.

Не использовать:

- Severity;
- Серьезность.

Варианты значений:

- Info — Информация;
- Warning — Предупреждение;
- Error — Ошибка;
- Critical — Критическая.

## Priority

Использовать:

    Приоритет

Варианты значений:

- Low — Низкий;
- Medium — Средний;
- High — Высокий;
- Critical — Критический.

## State

Использовать:

    Состояние

Примеры:

- Состояние аварии;
- Активна;
- Квитирована;
- Закрыта.

# Common UI states

## Loading

Использовать:

    Загрузка...

Для конкретных страниц:

- Загрузка данных...
- Загрузка текущих значений...
- Загрузка событий...
- Загрузка аварий...

## Empty

Использовать:

    Нет данных для отображения.

Для конкретных страниц:

- Нет текущих значений.
- Нет событий.
- Нет активных аварий.
- Нет устройств.
- Нет тегов.

## Error

Использовать:

    Не удалось загрузить данные.

Дополнительные варианты:

- Проверьте подключение к backend.
- Повторите попытку позже.
- Ошибка запроса к API.

## Mock data

Использовать:

    Демо-данные

или:

    Временные данные

Для предупреждений:

    На этой странице пока отображаются демо-данные. Реальное подключение к backend API будет добавлено позже.

Не использовать:

- Mock data;
- Mock values;
- Mock events;
- Mock alarms.

## Not implemented

Использовать:

    Функция пока не реализована.

или:

    Будет добавлено позже.

# Operator UI tone

Стиль интерфейса:

- короткий;
- нейтральный;
- технически точный;
- без разговорных формулировок;
- без лишнего маркетинга.

Примеры хорошего стиля:

- Нет активных аварий.
- Не удалось загрузить текущие значения.
- Данные обновлены.
- Требуется квитирование.
- Устройство недоступно.

Примеры плохого стиля:

- Ой, что-то пошло не так.
- Всё отлично!
- Данные магически обновились.
- Алёрты подъехали.
