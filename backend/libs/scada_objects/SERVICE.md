# scada_objects

## Назначение

`scada_objects` — backend-модуль объектной модели Dispatcher.

Модуль отвечает за универсальное дерево объектов, местоположение оборудования, логическую структуру диспетчеризируемого объекта и зоны ответственности.

Объектная модель является основой для:

- устройств;
- тегов;
- аварий;
- событий;
- дашбордов;
- мнемосхем;
- прав доступа;
- зон ответственности;
- ППР.

## Границы ответственности

`scada_objects` отвечает за:

- `ObjectId`;
- `ResponsibilityZoneId`;
- `ObjectNode`;
- `ObjectType`;
- `ObjectTree`;
- `ResponsibilityZone`;
- `IObjectRepository`;
- `IResponsibilityZoneRepository`;
- правила допустимой иерархии объектов;
- иерархию объектов;
- построение пути объекта;
- поиск объекта по идентификатору;
- получение корневых объектов;
- получение дочерних объектов;
- базовую проверку корректности дерева;
- базовую модель зоны ответственности;
- repository-интерфейсы объектной модели;
- подготовку связей для будущих устройств, оборудования, пользователей, ролей и прав доступа.

## Что модуль не должен делать

`scada_objects` не должен содержать:

- протокольные драйверы;
- опрос оборудования;
- текущие значения тегов;
- историю тегов;
- аварийную логику;
- команды управления;
- бизнес-логику устройств;
- полноценную модель пользователей;
- полноценную модель ролей;
- проверку прав доступа;
- SQL-запросы;
- PostgreSQL-реализацию;
- работу с PostgreSQL напрямую;
- UI-логику.

## Основные типы объектов

Поддерживаются базовые типы:

- `Site`;
- `Building`;
- `Floor`;
- `Room`;
- `Zone`;
- `Cabinet`;
- `Rack`;
- `Line`;
- `EquipmentGroup`;
- `Custom`.

## Типизированные идентификаторы

На текущем этапе добавлены:

- `ObjectId`;
- `ResponsibilityZoneId`.

Они построены поверх общего шаблона `dispatcher::common::EntityId<Tag>`.

Это нужно, чтобы не смешивать разные типы идентификаторов в коде.

Например:

    ObjectId object_id;
    ResponsibilityZoneId zone_id;

Такой подход позже будет расширен для устройств, тегов, аварий, событий и пользователей.

## Базовые правила иерархии

Корневыми объектами могут быть:

- `Site`;
- `Building`;
- `Zone`;
- `Line`;
- `EquipmentGroup`;
- `Custom`.

Пример допустимой иерархии 1:

    Site
     └── Building
          └── Floor
               └── Room
                    └── Cabinet
                         └── Rack

Пример допустимой иерархии 2:

    Site
     └── Zone
          └── Line
               └── EquipmentGroup

Пример допустимой иерархии 3:

    Building
     └── Room
          └── Cabinet

`Custom` разрешен как гибкий тип для случаев, когда стандартных типов недостаточно.

## Зоны ответственности

`ResponsibilityZone` описывает логическую область, за которую отвечают пользователи или роли.

Зона ответственности связывает:

- объекты;
- пользователей;
- роли.

Пример:

    ResponsibilityZone: PumpStationOperators

    Objects:
    - PumpStation01
    - Pump01
    - Pump02

    Users:
    - Operator01
    - Operator02

    Roles:
    - Operator

На текущем этапе `ResponsibilityZone` не проверяет права доступа и не зависит от будущего модуля пользователей.

Она только хранит связи, которые позже будут использоваться модулями:

- `scada_auth`;
- `scada_access_control`;
- `scada_alarms`;
- `scada_dashboards`;
- `scada_maintenance`.

## Repository-интерфейсы

На текущем этапе добавлены repository-интерфейсы.

### IObjectRepository

Интерфейс хранения объектной модели.

Отвечает за:

- получение всех объектов;
- поиск объекта по `ObjectId`;
- получение дочерних объектов;
- загрузку `ObjectTree`;
- сохранение объекта;
- удаление объекта.

### IResponsibilityZoneRepository

Интерфейс хранения зон ответственности.

Отвечает за:

- получение всех зон ответственности;
- поиск зоны по `ResponsibilityZoneId`;
- поиск зон по `ObjectId`;
- сохранение зоны;
- удаление зоны.

Repository-интерфейсы не содержат SQL и не знают, где именно хранятся данные.

Будущие реализации могут быть разными:

- in-memory repository для тестов;
- PostgreSQL repository для основной системы;
- file-based repository для отладки;
- mock repository для unit-тестов.
